# v3.2 Input Sync And Window Order Implementation Plan

> **For agentic workers:** REQUIRED: Use $subagent-driven-development (if subagents available) or $executing-plans to implement this plan. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** v3.2 只解决三个核心体验问题：键盘/鼠标同步可独立开关、鼠标离开主控窗口后不再继续触发受控窗口鼠标事件、窗口/进程列表排序稳定可预期。

**Architecture:** 保持当前纯 Win32 + Direct2D 自绘架构，不引入第三方依赖。同步引擎拆分键盘与鼠标转发判断，UI 增加两个轻量模式按钮，窗口枚举后统一走稳定排序规则，避免依赖 `EnumWindows` 的系统返回顺序。

**Tech Stack:** C++17, Win32 API, Direct2D, DirectWrite, low-level keyboard/mouse hooks, MSVC build.bat, Windows 10/11.

---

## Chunk 1: 设计边界与根因结论

### v3.2 范围

- 解决键盘同步和鼠标同步不能单独控制的问题。
- 解决窗口模式下鼠标移出主控窗口后，受控窗口仍收到鼠标事件的问题。
- 解决进程/窗口列表顺序不稳定，找主控窗口费劲的问题。
- 不改输入注入模型：继续使用 `WH_KEYBOARD_LL` / `WH_MOUSE_LL` 捕获，继续用 `PostMessageW` 向受控窗口投递消息。
- 不做更大的配置持久化、快捷键系统、窗口分组、拖拽排序等新功能，避免 v3.2 范围扩散。

### 根因调查

**问题 1：键盘和鼠标同步不能单独开关**

- 当前位置：`src/sync_engine.h`, `src/sync_engine.cpp`, `src/app_window.h`, `src/app_window.cpp`
- 现状：`SyncEngine::Start()` 同时安装键盘钩子和鼠标钩子；钩子回调都只判断 `m_running && ShouldForward()`。
- 根因：同步开关只有一个总状态 `m_running`，没有键盘/鼠标两个独立的功能状态。
- 设计方向：在 `SyncEngine` 中增加 `m_keyboardEnabled` 与 `m_mouseEnabled`，UI 增加“键盘同步”和“鼠标同步”两个模式按钮。总同步运行时，用户可独立开关两类事件转发。

**问题 2：窗口模式主控窗口移出后，受控窗口还会触发鼠标事件**

- 当前位置：`src/sync_engine.cpp`
- 现状：`ShouldForward()` 只判断 `GetForegroundWindow() == m_parent`。
- 根因：鼠标转发没有判断物理鼠标坐标是否仍在主控窗口客户区内。窗口模式下主控窗口仍可能是前台窗口，但光标已经移到任务栏、桌面、旁边窗口或其他区域，鼠标钩子仍继续把事件投递给受控窗口。
- 设计方向：保留键盘判断为“前台是主控窗口”，鼠标判断升级为“前台是主控窗口，并且鼠标屏幕坐标位于主控窗口客户区”。鼠标按钮、滚轮、移动都使用同一套边界判断。

**问题 3：进程排序随机，找主控窗口困难**

- 当前位置：`src/window_manager.cpp`, `src/app_window.cpp`
- 现状：`WindowManager::Refresh()` 直接使用 `EnumWindows` 收集结果；`ApplyFilter()` 按该顺序显示。
- 根因：`EnumWindows` 返回的是系统 Z-order/内部枚举顺序，不适合当作稳定 UI 排序。窗口打开、激活、刷新后顺序可能变化。
- 设计方向：枚举完成后执行稳定排序：进程名升序、窗口标题升序、PID 升序、HWND 数值兜底。过滤结果沿用已排序列表。

    assert(windows[2].title == L"A Window" && reinterpret_cast<uintptr_t>(windows[2].hWnd) == 100);

### Task 4: 修复主控窗口鼠标越界仍转发

- [ ] **Step 1: 实现主控客户区命中判断**

`PointInParentClient` 使用屏幕坐标输入，避免每个鼠标事件重复写转换逻辑：

```cpp
bool SyncEngine::PointInParentClient(int screenX, int screenY) const {
    if (!m_parent || !IsWindow(m_parent)) return false;

    RECT client = {};
    if (!GetClientRect(m_parent, &client)) return false;

    POINT origin = { 0, 0 };
    if (!ClientToScreen(m_parent, &origin)) return false;

    RECT screenClient = {
        origin.x,
        origin.y,
        origin.x + (client.right - client.left),
        origin.y + (client.bottom - client.top)
    };

    return screenX >= screenClient.left && screenX < screenClient.right &&
           screenY >= screenClient.top && screenY < screenClient.bottom;
}
```

- [ ] **Step 2: 拆分转发判断**

```cpp
bool SyncEngine::ShouldForwardKeyboard() const {
    if (!m_keyboardEnabled) return false;
    if (!m_parent || !IsWindow(m_parent)) return false;
    return GetForegroundWindow() == m_parent;
}

bool SyncEngine::ShouldForwardMouse(int screenX, int screenY) const {
    if (!m_mouseEnabled) return false;
    if (!m_parent || !IsWindow(m_parent)) return false;
    if (GetForegroundWindow() != m_parent) return false;
    return PointInParentClient(screenX, screenY);
}
```

## 风险与注意事项

- 鼠标边界应使用主控窗口客户区，不使用整个窗口矩形。否则鼠标在标题栏、边框时仍会误触发受控窗口。
- 键盘同步不应使用鼠标位置判断，否则用户鼠标短暂离开主控窗口时键盘同步会被意外切断。
- 不建议为键盘/鼠标切换动态安装/卸载钩子。保持钩子生命周期跟随 `Start()` / `Stop()`，只在回调中判断模式，逻辑更简单，失败面更小。
- 排序不要放在 UI 过滤之后临时做。应在 `WindowManager` 枚举层统一排序，让父窗口列表和子窗口列表天然一致。
- UI 下方已经修过 Win10 DPI 裁切，v3.2 action bar 加按钮时必须优先保证 860x620 默认窗口完整显示。
- v3.2 不引入配置持久化；键盘/鼠标开关默认都为开启，符合 v3.1 用户习惯。

## Definition Of Done

- `tests\build_tests.bat` 通过。
- `cmd /c "echo. | build.bat"` 成功生成 `dx11_sync.exe`。
- Win10 下底部组件仍完整显示。
- 键盘/鼠标同步可以独立开关。
- 鼠标离开主控窗口客户区后，受控窗口不再触发鼠标事件。
- 窗口列表刷新后排序稳定。
- 版本显示为 v3.2。
- 代码提交并推送到 GitHub `master`。
